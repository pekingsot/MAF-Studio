# 协作工作流设计文档

## 1. 任务与工作流的关系

### 1.1 概念澄清

**协作任务（CollaborationTask）**：
- 用于跟踪和管理协作中的具体任务
- 包含任务状态、分配、进度等信息
- 是工作流执行的**输入来源**

**工作流（Workflow）**：
- 定义任务如何被多个Agent处理
- 包含执行模式、参数配置等
- 是任务执行的**处理方式**

### 1.2 推荐的使用流程

```
1. 创建协作任务
   POST /api/collaborations/{id}/tasks
   {
     "title": "设计登录页面",
     "description": "设计一个现代化的登录页面"
   }
   
2. 选择任务执行工作流
   POST /api/collaborations/{id}/workflow/execute
   {
     "taskId": 1001,
     "workflowType": "Sequential",
     "parameters": {
       "maxRounds": 5,
       "timeout": 300
     }
   }
   
3. 查看任务执行结果
   GET /api/collaborations/{id}/tasks/{taskId}
```

## 2. 四种工作流模式详解

### 2.1 顺序执行

**适用场景**：
- 需要按步骤处理的任务
- 每个Agent的输出是下一个Agent的输入
- 例如：需求分析 → 设计 → 开发 → 测试

**执行流程**：
```
用户输入 → Agent1 → Agent2 → Agent3 → 最终结果
```

**配置参数**：
- `agents`: 按顺序指定Agent列表
- `stopCondition`: 停止条件（如包含特定关键词）

**示例**：
```json
{
  "taskId": 1001,
  "workflowType": "Sequential",
  "parameters": {
    "agentOrder": [1, 2, 3],
    "stopKeywords": ["完成", "DONE"]
  }
}
```

### 2.2 并发执行

**适用场景**：
- 多个Agent独立处理同一任务
- 需要多个视角或方案
- 例如：多个设计师同时设计不同方案

**执行流程**：
```
用户输入 → ┬→ Agent1 → 结果1
          ├→ Agent2 → 结果2
          └→ Agent3 → 结果3
          
最终结果 = 汇总所有结果
```

**配置参数**：
- `agents`: 参与的Agent列表
- `aggregationStrategy`: 结果聚合策略（投票、合并、选择最佳等）
- `maxConcurrency`: 最大并发数

**示例**：
```json
{
  "taskId": 1001,
  "workflowType": "Concurrent",
  "parameters": {
    "agents": [1, 2, 3],
    "aggregationStrategy": "Merge",
    "maxConcurrency": 3
  }
}
```

### 2.3 任务移交

**适用场景**：
- Agent之间需要协作和交接
- 每个Agent处理自己擅长的部分
- 例如：前端开发 → 后端开发 → 测试

**执行流程**：
```
用户输入 → Agent1处理 → 判断是否需要移交
                            ↓
                        [HANDOFF:Agent2]
                            ↓
                        Agent2处理 → 判断是否需要移交
                                        ↓
                                    [HANDOFF:Agent3]
                                        ↓
                                    Agent3处理 → 完成
```

**配置参数**：
- `agents`: 参与的Agent列表
- `handoffKeyword`: 移交关键词（默认：`[HANDOFF:agent_name]`）
- `maxHandoffs`: 最大移交次数

**示例**：
```json
{
  "taskId": 1001,
  "workflowType": "Handoffs",
  "parameters": {
    "agents": [1, 2, 3],
    "handoffKeyword": "[HANDOFF:",
    "maxHandoffs": 10
  }
}
```

### 2.4 群聊协作

**适用场景**：
- 需要多Agent讨论和协作
- 复杂问题的多轮对话
- 例如：头脑风暴、方案评审

**执行流程**：
```
用户输入 → Agent1发言 → Agent2发言 → Agent3发言
    ↓           ↓            ↓            ↓
  轮次1      轮次1        轮次1        轮次1
    ↓           ↓            ↓            ↓
  轮次2      轮次2        轮次2        轮次2
    ↓           ↓            ↓            ↓
  ...         ...          ...          ...
    ↓
  [END] 结束
```

**配置参数**：
- `agents`: 参与的Agent列表
- `maxRounds`: 最大轮次
- `endKeyword`: 结束关键词（默认：`[END]`）
- `moderatorAgent`: 主持人Agent（可选）

**示例**：
```json
{
  "taskId": 1001,
  "workflowType": "GroupChat",
  "parameters": {
    "agents": [1, 2, 3],
    "maxRounds": 10,
    "endKeyword": "[END]",
    "moderatorAgent": 1
  }
}
```

## 3. 工作流配置建议

### 3.1 协作级别配置

在创建协作时，可以配置默认工作流：

```json
POST /api/collaborations
{
  "name": "UI设计协作",
  "description": "UI设计多Agent协作",
  "defaultWorkflow": {
    "type": "Sequential",
    "parameters": {
      "agentOrder": [1, 2, 3],
      "stopKeywords": ["完成"]
    }
  }
}
```

### 3.2 任务级别配置

在创建任务时，可以指定工作流：

```json
POST /api/collaborations/{id}/tasks
{
  "title": "设计登录页面",
  "description": "设计一个现代化的登录页面",
  "workflowType": "Concurrent",
  "workflowParameters": {
    "agents": [1, 2, 3],
    "aggregationStrategy": "Merge"
  }
}
```

### 3.3 Agent级别配置

在添加Agent到协作时，可以配置角色和能力：

```json
POST /api/collaborations/{id}/agents
{
  "agentId": 1,
  "role": "UI设计师",
  "capabilities": ["设计", "评审"],
  "workflowPreferences": {
    "preferredPosition": "first",
    "canHandoffTo": [2, 3]
  }
}
```

## 4. 实现建议

### 4.1 数据库表结构优化

```sql
-- 协作工作流配置表
CREATE TABLE collaboration_workflows (
    id BIGSERIAL PRIMARY KEY,
    collaboration_id BIGINT NOT NULL REFERENCES collaborations(id),
    name VARCHAR(100) NOT NULL,
    type VARCHAR(50) NOT NULL,
    parameters JSONB,
    is_default BOOLEAN DEFAULT false,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 任务执行记录表
CREATE TABLE task_executions (
    id BIGSERIAL PRIMARY KEY,
    task_id BIGINT NOT NULL REFERENCES collaboration_tasks(id),
    workflow_id BIGINT REFERENCES collaboration_workflows(id),
    status VARCHAR(50) NOT NULL,
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    result TEXT,
    error TEXT
);

-- Agent执行记录表
CREATE TABLE agent_executions (
    id BIGSERIAL PRIMARY KEY,
    execution_id BIGINT NOT NULL REFERENCES task_executions(id),
    agent_id BIGINT NOT NULL,
    input TEXT,
    output TEXT,
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    status VARCHAR(50)
);
```

### 4.2 API设计

```csharp
// 执行工作流
POST /api/collaborations/{id}/workflow/execute
{
    "taskId": 1001,
    "workflowType": "Sequential",
    "parameters": { ... }
}

// 获取工作流配置
GET /api/collaborations/{id}/workflows

// 创建工作流配置
POST /api/collaborations/{id}/workflows
{
    "name": "默认顺序流程",
    "type": "Sequential",
    "parameters": { ... }
}

// 获取执行历史
GET /api/collaborations/{id}/executions
```

## 6. 总结

### 核心改进点：

1. **明确任务与工作流的关系**
   - 任务是工作流的输入
   - 工作流是任务的处理方式

2. **提供灵活的配置选项**
   - 协作级别默认配置
   - 任务级别特定配置
   - Agent级别能力配置

3. **可视化工作流设计**
   - 图形化配置界面
   - 实时预览执行流程
   - 执行历史追踪

4. **完善的数据记录**
   - 任务执行记录
   - Agent执行记录
   - 结果分析和统计
