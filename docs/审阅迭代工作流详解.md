# 审阅迭代工作流详解

## 📋 场景描述

**典型场景**：
```
A写文档 → B审阅 → 不满意 → 打回去 → A修改 → B审阅 → ... → 满意 → 结束
```

这是一个非常常见的**迭代审阅流程**，适用于：
- 文档编写与审阅
- 代码开发与Code Review
- 设计方案与评审
- 方案制定与审批

---

## 🎯 为什么需要专门的工作流？

### 现有工作流的局限性

| 工作流类型 | 是否适合 | 问题 |
|-----------|---------|------|
| **顺序执行** | ❌ | 只能执行一次，无法迭代 |
| **并发执行** | ❌ | 同时执行，无法审阅 |
| **任务移交** | ⚠️ | 可以实现，但需要Agent自己判断何时移交 |
| **群聊协作** | ✅ | 可以实现，但需要手动控制结束 |

### 专门设计的优势

**审阅迭代工作流**专门为此场景设计：

```
┌─────────────────────────────────────────┐
│  迭代1                                   │
│  ┌──────────┐      ┌──────────┐         │
│  │ A编写文档 │ ──→ │ B审阅文档 │         │
│  └──────────┘      └─────┬────┘         │
│                          │               │
│                          ↓ 不满意        │
│                     打回去修改            │
└─────────────────────────────────────────┘
           ↓
┌─────────────────────────────────────────┐
│  迭代2                                   │
│  ┌──────────┐      ┌──────────┐         │
│  │ A修改文档 │ ──→ │ B审阅文档 │         │
│  └──────────┘      └─────┬────┘         │
│                          │               │
│                          ↓ 满意          │
│                     [APPROVED]           │
└─────────────────────────────────────────┘
           ↓
        流程结束
```

## 🚀 使用方式

### API调用

#### 方式1：使用专用接口

```bash
POST /api/collaborations/{collaborationId}/workflow/review-iterative
{
  "input": "编写一份系统架构设计文档",
  "maxIterations": 10,
  "reviewCriteria": "请从架构合理性、可扩展性、安全性等方面进行审阅",
  "approvalKeyword": "[APPROVED]",
  "saveVersions": true
}
```

#### 方式2：使用通用接口

```bash
POST /api/collaborations/{collaborationId}/workflow/execute
{
  "workflowType": "ReviewIterative",
  "input": "编写一份系统架构设计文档",
  "parameters": {
    "maxIterations": 10,
    "reviewCriteria": "请从架构合理性、可扩展性、安全性等方面进行审阅"
  }
}
```

### 参数说明

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| **input** | string | 必填 | 初始任务描述 |
| **maxIterations** | int | 10 | 最大迭代次数（防止无限循环） |
| **reviewCriteria** | string | 可选 | 审阅标准，指导审阅者如何审阅 |
| **approvalKeyword** | string | "[APPROVED]" | 审阅通过的关键词 |
| **saveVersions** | bool | true | 是否保存每次迭代的版本 |

## 📊 执行流程详解

### 完整执行流程

```
开始
  ↓
迭代次数 = 0
  ↓
┌─────────────────────────────────────┐
│ 迭代循环（iteration < maxIterations）│
│                                      │
│  1. 编写者创建/修改内容               │
│     - 第1次：根据初始要求创建         │
│     - 后续：根据审阅意见修改          │
│                                      │
│  2. 审阅者审阅内容                    │
│     - 检查是否符合要求                │
│     - 给出具体反馈                    │
│                                      │
│  3. 判断是否通过                      │
│     - 包含 [APPROVED] → 通过，结束    │
│     - 不包含 → 继续，进入下一轮       │
│                                      │
│  4. iteration++                      │
└─────────────────────────────────────┘
  ↓
结束（通过或达到最大迭代次数）
```

### 详细示例

#### 第1轮迭代

**编写者（A）**：
```
输入：请根据以下要求创建文档：编写一份系统架构设计文档

输出：
# 系统架构设计文档

## 1. 系统概述
本系统采用微服务架构...

## 2. 技术选型
- 后端：.NET 10
- 数据库：PostgreSQL
...
```

**审阅者（B）**：
```
输入：请审阅以下文档...

输出：
审阅意见：

1. 架构设计部分缺少高可用方案
2. 数据库选型需要考虑读写分离
3. 缺少安全设计章节

请修改以上问题。
```

**判断**：不包含 `[APPROVED]`，继续下一轮

---

#### 第2轮迭代

**编写者（A）**：
```
输入：请根据审阅意见修改文档...

输出：
# 系统架构设计文档

## 1. 系统概述
本系统采用微服务架构...

## 2. 高可用设计
- 服务注册与发现
- 负载均衡
- 故障转移...

## 3. 数据库设计
- 主从复制
- 读写分离...

## 4. 安全设计
- 认证授权
- 数据加密...
```

**审阅者（B）**：
```
输入：请审阅以下文档...

输出：
文档已完善，符合要求。

[APPROVED]
```

**判断**：包含 `[APPROVED]`，审阅通过，流程结束

## 💡 最佳实践

### 1. 设置合理的最大迭代次数

```
简单文档：3-5 次
中等复杂：5-10 次
复杂项目：10-20 次
```

### 2. 提供清晰的审阅标准

**好的审阅标准**：
```
请从以下方面进行审阅：
1. 内容完整性：是否涵盖所有必要章节
2. 逻辑性：论述是否清晰、有条理
3. 准确性：技术描述是否准确
4. 格式规范：是否符合文档规范
```

**不好的审阅标准**：
```
请审阅文档
```

### 3. 使用明确的通过关键词

推荐使用：
- `[APPROVED]` - 标准关键词
- `[审阅通过]` - 中文关键词
- `[OK]` - 简短关键词

### 4. 保存版本历史

启用 `saveVersions: true` 可以：
- 查看每次迭代的变化
- 回退到之前的版本
- 分析改进过程

## 🔍 与其他工作流的对比

| 特性 | 审阅迭代 | 群聊协作 | 任务移交 |
|------|---------|---------|---------|
| **迭代支持** | ✅ 专门设计 | ✅ 支持 | ⚠️ 需手动控制 |
| **角色固定** | ✅ 编写者/审阅者 | ❌ 所有Agent平等 | ⚠️ 需要Agent判断 |
| **结束条件** | ✅ 自动判断 | ⚠️ 需手动结束 | ⚠️ 需Agent判断 |
| **反馈机制** | ✅ 结构化反馈 | ✅ 自由讨论 | ⚠️ 需Agent理解 |
| **适用场景** | 文档审阅、代码Review | 头脑风暴、方案讨论 | 任务流转 |

## 🎯 总结

### 适用场景

✅ **强烈推荐**：
- 文档编写与审阅
- 代码开发与Code Review
- 设计方案与评审
- 方案制定与审批

⚠️ **可以考虑**：
- 需求分析与确认
- 测试用例编写与审核

❌ **不推荐**：
- 简单的一次性任务
- 不需要审阅的任务
- 需要多方讨论的复杂问题（建议用群聊协作）

### 核心优势

1. **自动化迭代** - 无需手动控制，自动循环直到满意
2. **结构化反馈** - 审阅者给出明确的修改意见
3. **防止无限循环** - 通过最大迭代次数限制
4. **版本追踪** - 可以查看每次迭代的变化
5. **灵活配置** - 可以自定义审阅标准和通过条件
