# 协作工作流问题解答

## 问题1：协作工作流的任务到底执行哪个任务？

### ❌ 当前问题

目前系统中存在**概念混淆**：

1. **协作任务（collaboration_tasks）** - 用于跟踪任务状态
2. **工作流执行** - 直接接收输入字符串，与任务无关

这导致用户困惑：**我创建的任务和工作流执行有什么关系？**

### ✅ 正确的设计

**任务与工作流的关系**：

```
┌─────────────┐
│  创建任务    │  ← 记录要做什么
│  (What)     │
└──────┬──────┘
       │
       ↓
┌─────────────┐
│  选择工作流  │  ← 决定怎么做
│  (How)      │
└──────┬──────┘
       │
       ↓
┌─────────────┐
│  执行工作流  │  ← 实际执行
│  (Execute)  │
└─────────────┘
```

**推荐的使用流程**：

#### 方式1：从任务执行工作流（推荐）

```
1. 创建任务
   POST /api/collaborations/{id}/tasks
   {
     "title": "设计登录页面",
     "description": "设计一个现代化的登录页面"
   }
   
2. 选择任务执行工作流
   POST /api/collaborations/{id}/workflow/execute
   {
     "taskId": 1001,  ← 关联任务
     "workflowType": "Sequential",
     "parameters": { ... }
   }
   
3. 查看任务执行结果
   GET /api/collaborations/{id}/tasks/{taskId}
```

#### 方式2：直接执行工作流（快速测试）

```
POST /api/collaborations/{id}/workflow/execute
{
  "workflowType": "Sequential",
  "input": "帮我设计一个登录页面"  ← 直接输入
}
```

### 📊 对比

| 方式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **从任务执行** | 正式工作流程 | 有记录、可追踪、可统计 | 步骤多 |
| **直接执行** | 快速测试、临时任务 | 简单快速 | 无记录 |

## 问题2：四种工作流模式如何流转？

### 2.1 顺序执行（Sequential）

**执行流程**：

```
输入 → Agent1 → Agent2 → Agent3 → 输出
       (分析)    (设计)    (实现)
```

**特点**：
- ✅ 按顺序执行，前一个的输出是后一个的输入
- ✅ 适合流水线式任务
- ❌ 不能并行，速度较慢

**配置方式**：

```json
{
  "workflowType": "Sequential",
  "parameters": {
    "agentOrder": [1, 2, 3],  // 手动指定顺序
    "stopKeywords": ["完成", "DONE"]  // 遇到这些词停止
  }
}
```

**Agent如何处理**：
- Agent按配置的顺序依次执行
- 不需要Agent自己判断，系统自动流转

---

### 2.2 并发执行（Concurrent）

**执行流程**：

```
       ┌→ Agent1 → 结果1 ┐
输入 → ├→ Agent2 → 结果2 ├→ 汇总 → 输出
       └→ Agent3 → 结果3 ┘
```

**特点**：
- ✅ 多个Agent同时工作，速度快
- ✅ 可以得到多个方案
- ❌ 需要汇总策略

**配置方式**：

```json
{
  "workflowType": "Concurrent",
  "parameters": {
    "agents": [1, 2, 3],  // 参与的Agent
    "aggregationStrategy": "Merge",  // 汇总策略
    "maxConcurrency": 3  // 最大并发数
  }
}
```

**汇总策略**：
- **Merge** - 合并所有结果（默认）
- **Vote** - 投票选择最佳
- **First** - 使用第一个完成的结果
- **Last** - 使用最后一个完成的结果

**Agent如何处理**：
- 系统同时启动所有Agent
- Agent独立处理，互不干扰
- 系统自动汇总结果

---

### 2.3 任务移交（Handoffs）

**执行流程**：

```
输入 → Agent1处理
         ↓
      判断：需要移交吗？
         ↓ 是
      [HANDOFF:Agent2]  ← Agent1输出这个标记
         ↓
      Agent2处理
         ↓
      判断：需要移交吗？
         ↓ 是
      [HANDOFF:Agent3]  ← Agent2输出这个标记
         ↓
      Agent3处理
         ↓
      完成
```

**特点**：
- ✅ Agent可以自主决定移交给谁
- ✅ 灵活性高，适合复杂协作
- ❌ 需要Agent理解移交机制

**配置方式**：

```json
{
  "workflowType": "Handoffs",
  "parameters": {
    "agents": [1, 2, 3],
    "handoffKeyword": "[HANDOFF:",  // 移交标记
    "maxHandoffs": 10  // 最大移交次数（防止死循环）
  }
}
```

**Agent如何处理**：
- Agent在输出中包含 `[HANDOFF:AgentName]` 来移交
- 系统解析这个标记，自动路由到下一个Agent
- 如果没有移交标记，则顺序执行下一个

**示例**：

```
Agent1 (UI设计师):
"我已经完成了UI设计，现在移交给开发团队实现。
[HANDOFF:前端开发]"

系统解析后 → 自动转给"前端开发"Agent
```

---

### 2.4 群聊协作（GroupChat）

**执行流程**：

```
轮次1: 用户 → Agent1 → Agent2 → Agent3
轮次2: Agent1 → Agent2 → Agent3 → Agent1
轮次3: Agent2 → Agent3 → Agent1 → Agent2
...
直到有人输出 [END] 或达到最大轮次
```

**特点**：
- ✅ 多Agent讨论，适合复杂问题
- ✅ 可以多轮对话，深入探讨
- ❌ 可能会无限循环，需要限制轮次

**配置方式**：

```json
{
  "workflowType": "GroupChat",
  "parameters": {
    "agents": [1, 2, 3],
    "maxRounds": 10,  // 最大轮次（防止无限循环）
    "endKeyword": "[END]"  // 结束标记
  }
}
```

**Agent如何处理**：
- 系统按轮次让每个Agent发言
- Agent可以看到之前所有Agent的发言
- Agent输出 `[END]` 来结束讨论

**示例**：

```
轮次1:
- Agent1: "我认为应该使用React框架..."
- Agent2: "我同意，但需要考虑性能优化..."
- Agent3: "建议使用Next.js，它有更好的SSR支持..."

轮次2:
- Agent1: "好的，那我们决定使用Next.js [END]"
- (讨论结束)
```

## 🎯 总结对比

| 工作流类型 | 执行方式 | Agent自主性 | 适用场景 | 配置复杂度 |
|-----------|---------|------------|---------|-----------|
| **顺序执行** | 系统控制流转 | 低 | 流水线任务 | ⭐ 简单 |
| **并发执行** | 系统并行启动 | 低 | 需要多个方案 | ⭐⭐ 中等 |
| **任务移交** | Agent决定移交 | 高 | 复杂协作 | ⭐⭐⭐ 复杂 |
| **群聊协作** | Agent多轮讨论 | 高 | 复杂问题讨论 | ⭐⭐ 中等 |

## 🚀 实施建议

### 第一步：明确使用场景

```
顺序执行 → 需求分析 → 设计 → 开发 → 测试
并发执行 → 多个设计师同时设计不同方案
任务移交 → 前端开发 → 后端开发 → 测试（按需移交）
群聊协作 → 技术方案评审、头脑风暴
```

### 第二步：配置工作流

在前端界面提供可视化配置：

```
┌─────────────────────────────────┐
│  选择工作流类型                  │
│  ● 顺序执行  ○ 并发执行          │
│  ○ 任务移交  ○ 群聊协作          │
│                                  │
│  Agent顺序：                     │
│  [↑] Agent1 - 需求分析师         │
│  [↑] Agent2 - UI设计师           │
│  [↑] Agent3 - 前端开发           │
│                                  │
│  [执行工作流]                    │
└─────────────────────────────────┘
```

## 📝 快速开始

### 最简单的使用方式

```bash
# 1. 创建协作
POST /api/collaborations
{
  "name": "UI设计协作"
}

# 2. 添加Agent
POST /api/collaborations/{id}/agents
{
  "agentId": 1,
  "role": "UI设计师"
}

# 3. 执行工作流
POST /api/collaborations/{id}/workflow/execute
{
  "workflowType": "Sequential",
  "input": "设计一个登录页面"
}
```

就这么简单！🎉
