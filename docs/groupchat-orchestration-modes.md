# GroupChat协调模式详解

## 概述

MAF Studio的GroupChat工作流支持三种协调模式，每种模式适用于不同的场景。本文档详细解释每种模式的工作原理、适用场景和最佳实践。

---

## 三种协调模式

### 1. RoundRobin（轮询模式） 🔄

#### 工作原理
- **固定顺序**：按照Agent添加的顺序，依次让每个Agent发言
- **无协调者**：所有Agent地位平等，没有领导者
- **简单公平**：每个Agent都有均等的发言机会

#### 执行流程
```
轮次1: Agent1 → Agent2 → Agent3
轮次2: Agent1 → Agent2 → Agent3
轮次3: Agent1 → Agent2 → Agent3
...
```

#### 适用场景
- ✅ 所有Agent能力相当，没有明显的领导者
- ✅ 任务简单，不需要复杂的协调和调度
- ✅ 需要快速收集所有Agent的意见
- ✅ 团队成员之间平等协作

#### 优点
- 实现简单，易于理解
- 公平性高，每个Agent都有发言机会
- 不需要额外的协调者角色
- 执行速度快，没有额外的协调开销

#### 缺点
- 缺乏智能调度，可能让不合适的Agent发言
- 无法根据任务需求动态调整发言顺序
- 可能产生冗余对话，效率较低

#### 配置示例
```json
{
  "workflowType": "GroupChat",
  "orchestrationMode": "RoundRobin",
  "maxIterations": 10,
  "workerAgents": [
    { "agentId": 1001 },
    { "agentId": 1002 },
    { "agentId": 1003 }
  ]
}
```

---

### 2. Manager（管理者模式） 👔

#### 工作原理
- **有协调者**：需要一个Manager角色的Agent负责协调
- **智能调度**：Manager根据任务需求选择下一个发言者
- **@提及机制**：Manager可以通过@提及指定Agent发言
- **双向协调**：Worker发言后，回到Manager继续协调

#### 执行流程
```
Manager: "我需要@架构师来设计系统架构"
→ 架构师发言
→ Manager: "好的，现在让@开发工程师来实现"
→ 开发工程师发言
→ Manager: "让@测试工程师来验证"
→ 测试工程师发言
→ Manager: "任务完成"
```

#### 适用场景
- ✅ 需要统一协调的复杂任务
- ✅ 团队中有明确的领导者角色
- ✅ 需要根据上下文动态选择Agent
- ✅ 多步骤、多阶段的任务

#### 优点
- 智能调度，根据任务需求选择最合适的Agent
- 避免无效对话，提高效率
- 可以处理复杂的多步骤任务
- Manager可以总结和整合信息

#### 缺点
- 需要一个专门的Manager角色
- Manager的能力直接影响团队效率
- Manager可能成为瓶颈

#### 配置示例
```json
{
  "workflowType": "GroupChat",
  "orchestrationMode": "Manager",
  "managerAgentId": 1001,
  "managerCustomPrompt": "你是项目经理，负责协调团队成员...",
  "maxIterations": 10,
  "workerAgents": [
    { "agentId": 1002 },
    { "agentId": 1003 },
    { "agentId": 1004 }
  ]
}
```

#### Manager的最佳实践
1. **选择合适的Manager**：
   - Manager应该具备全局视角
   - Manager应该了解每个Worker的能力
   - Manager应该有良好的沟通和协调能力

2. **自定义Manager提示词**：
   ```
   你是项目经理，负责协调以下团队成员：
   - 架构师：负责系统架构设计
   - 开发工程师：负责代码实现
   - 测试工程师：负责质量验证
   
   你的职责：
   1. 分析任务需求，确定执行步骤
   2. 根据任务需求选择合适的团队成员
   3. 使用@提及指定发言者
   4. 整合团队成员的输出
   5. 判断任务是否完成
   ```

3. **Manager的决策逻辑**：
   - 检查消息中的@提及
   - 如果有@提及，选择被提及的Agent
   - 如果没有@提及，按轮询顺序选择Worker
   - Worker发言后，回到Manager继续协调

### 3. Intelligent（智能模式） 🧠

#### 工作原理
- **AI决策**：使用LLM模型选择下一个发言者
- **上下文分析**：分析对话历史和任务上下文
- **动态选择**：根据当前任务状态智能选择Agent
- **自适应**：能够根据任务进展调整策略

#### 执行流程
```
AI分析: "当前需要架构设计，选择架构师"
→ 架构师发言
→ AI分析: "架构已完成，需要实现，选择开发工程师"
→ 开发工程师发言
→ AI分析: "代码已实现，需要测试，选择测试工程师"
→ 测试工程师发言
→ AI分析: "测试通过，任务完成"
```

#### 适用场景
- ✅ 高度复杂的任务
- ✅ 需要AI判断最优发言者
- ✅ Agent能力差异较大
- ✅ 任务流程不确定，需要动态调整

#### 优点
- 最智能的调度方式
- 能够根据任务进展动态调整
- 适应性强，可以处理各种复杂情况
- 不需要专门的Manager角色

#### 缺点
- 消耗更多的token（每次选择都需要调用LLM）
- 决策速度较慢
- 可能出现选择错误
- 成本较高

#### 配置示例
```json
{
  "workflowType": "GroupChat",
  "orchestrationMode": "Intelligent",
  "maxIterations": 10,
  "workerAgents": [
    { "agentId": 1001 },
    { "agentId": 1002 },
    { "agentId": 1003 }
  ]
}
```

## 如何选择合适的模式？

### 决策树

```
开始
  │
  ├─ 团队中是否有Manager角色的Agent？
  │   ├─ 否 → 使用 RoundRobin
  │   └─ 是 → 继续
  │
  ├─ 任务复杂度如何？
  │   ├─ 简单 → 使用 RoundRobin
  │   ├─ 中等 → 使用 Manager
  │   └─ 复杂 → 使用 Intelligent
  │
  └─ 是否需要动态调整？
      ├─ 否 → 使用 Manager
      └─ 是 → 使用 Intelligent
```

### 推荐场景

#### 使用RoundRobin的场景
- 快速头脑风暴会议
- 收集所有团队成员的意见
- 简单的信息同步和分享
- 团队成员能力相当

#### 使用Manager的场景
- 软件开发项目（架构师→开发→测试）
- 需要明确分工的任务
- 多阶段、多步骤的工作流
- 有项目经理或技术负责人的团队

#### 使用Intelligent的场景
- 研究型项目，流程不确定
- 高度复杂的决策任务
- Agent能力差异较大
- 需要AI判断最优路径

## 最佳实践

### 1. 团队角色配置

#### RoundRobin模式
```
团队成员：
- Agent1（Worker）
- Agent2（Worker）
- Agent3（Worker）
```

#### Manager模式
```
团队成员：
- 项目经理（Manager）← 协调者
- 架构师（Worker）
- 开发工程师（Worker）
- 测试工程师（Worker）
```

#### Intelligent模式
```
团队成员：
- Agent1（Worker）
- Agent2（Worker）
- Agent3（Worker）
```

### 2. 提示词设计

#### RoundRobin模式
```
每个Agent的提示词应该：
- 明确自己的职责
- 了解其他Agent的能力
- 简洁高效地完成任务
```

#### Manager模式
```
Manager的提示词应该：
- 了解每个Worker的能力和职责
- 能够分析任务需求
- 使用@提及指定发言者
- 判断任务是否完成

Worker的提示词应该：
- 专注于自己的专业领域
- 清晰地表达自己的输出
- 不需要关心协调逻辑
```

#### Intelligent模式
```
每个Agent的提示词应该：
- 清晰描述自己的能力
- 明确自己的适用场景
- 提供足够的上下文信息
```

### 3. 性能优化

#### RoundRobin模式
- 设置合理的`maxIterations`，避免无限循环
- 监控对话质量，及时终止无效对话

#### Manager模式
- 优化Manager的提示词，提高决策准确性
- 监控Manager的选择逻辑，避免重复选择
- 设置明确的终止条件

#### Intelligent模式
- 控制对话历史长度，减少token消耗
- 缓存AI的决策结果，避免重复计算
- 监控AI的选择准确性，及时调整

## 常见问题

### Q1: 为什么我的Manager模式不工作？
**A**: 检查以下几点：
1. 是否正确设置了`managerAgentId`
2. Manager Agent是否有Manager角色
3. Manager的提示词是否正确配置

### Q2: Intelligent模式太慢怎么办？
**A**: 尝试以下优化：
1. 减少对话历史长度
2. 使用更快的LLM模型
3. 考虑切换到Manager模式

### Q3: 如何判断任务是否完成？
**A**: 三种模式有不同的终止条件：
- RoundRobin：达到`maxIterations`或检测到终止关键词
- Manager：Manager判断任务完成或达到`maxIterations`
- Intelligent：AI判断任务完成或达到`maxIterations`

### Q4: 可以混合使用多种模式吗？
**A**: 不可以，每个任务只能选择一种协调模式。但可以在不同任务中使用不同的模式。

---

## 总结

选择合适的协调模式是GroupChat工作流成功的关键：

- **RoundRobin**：简单、公平、快速，适合简单任务
- **Manager**：智能、高效、可控，适合中等复杂任务
- **Intelligent**：最智能、最灵活，适合高度复杂任务

根据团队配置、任务复杂度和性能要求，选择最适合的模式，能够显著提高任务执行效率和质量。
